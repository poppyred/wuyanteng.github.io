redis主从宕机数据恢复问题

2017-11-23 
今天对redis在生产使用过程中的数据恢复问题做个剖析。

前提：

适用于: redis持久化没有采用AOF的持久化方式，而是采用的RDB方式。因为AOF的方式会对硬盘io造成很大压力。

既然采用RDB方式持久化，那么数据会根据自定义save策略自动进行持久化，并保存数据到指定目录中，文件以rdb结尾。

情况1：从节点宕机

这种情况很简单，从节点宕机后，会丢失宕机时间内的数据，服务器维护好并启动后，会自动与Redis主进行数据同步。

情况2：集群主节点宕机

一个 redis 集群包含 16384 个哈希槽（hash slot），数据库中的每个数据都属于这16384个哈希槽中的一个。集群使用公式 CRC16(key) % 16384 来计算键 key 属于哪个槽。集群中的每一个节点负责处理一部分哈希槽

集群中的主从复制:
集群中的每个节点都有1个至N个复制品，其中一个为主节点，其余的为从节点，如果主节点下线了，集群就会把这个主节点的一个从节点设置为新的主节点，继续工作。这样集群就不会因为一个主节点的下线而无法正常工作

注意：
1、如果某一个主节点和他所有的从节点都下线的话，redis集群就会停止工作了。redis集群不保证数据的强一致性，在特定的情况下，redis集群会丢失已经被执行过的写命令
2、使用异步复制（asynchronous replication）是redis 集群可能会丢失写命令的其中一个原因，有时候由于网络原因，如果网络断开时间太长，redis集群就会启用新的主节点，之前发给主节点的数据就会丢失。

这种多主多从的架构一般不会造成数据丢失。除非主都挂了...

情况3：一主一丛

我要说的就是这种情况，redis主本地没有缓存数据，从redis负责数据落地。这样可以降低redis主服务器的压力。

当主节点挂掉，当维护完毕这台主服务器时，需要慎重启动redis服务（最好不要自启动），因为此时主宕机期间数据就已经丢失了，一旦启动服务从就会从主节点同步过时数据，造成数据丢失。

正确流程：
1. 修改主节点配置文件，修改端口（让从找不到主）
2. 在配置文件内加入如下内容（启用如下配置）：
   slaveof  10.10.10.8 6379  （把主临时改为10.10.10.8从节点的从节点）
   slave-serve-stale-data yes  

   注：slaveof选项其实就是配置redis主服务器和端口

3. 启动刚才宕机的主redis服务
4. redis-cli -h 10.10.10.10 -p 6379  #在主redis上，登陆redis命令界面
   执行save命令，同步10.10.10.8从节点的rdb数据到主上来，同步完成后，再注释掉刚才配置文件添加的两行配置，修改为主的端口，最后重新启动redis。

常用命令

#获取RDB文件保存路径：
redis 10.10.10.10:6379> CONFIG GET dir

#Redis SAVE 命令用于创建当前数据库的备份
redis 10.10.10.10:6379>SAVE

#创建 redis RDB备份文件也可以使用命令BGSAVE，该命令在后台执行
redis 10.10.10.10:6379> BGSAVE

#创建 redis 备份文件也可以使用命令 BGSAVE，该命令在后台执行。
实例
127.0.0.1:6379> BGSAVE
Background saving started
