Tomcat部署实例集群监控

2017-11-18

Tomcat部署

只要是java相关的应用必装JDK

安装JDK

tar xf jdk-8u60-linux-x64.tar.gz -C /application/

ln -s /application/jdk1.8.0_60 /application/jdk

sed -i.ori '$a export JAVA_HOME=/application/jdk\nexport PATH=$JAVA_HOME/bin:$JAVA_HOME/jre/bin:$PATH\nexport CLASSPATH=.:$JAVA_HOME/lib:$JAVA_HOME/jre/lib:$JAVA_HOME/lib/tools.jar' /etc/profile
注：$a表示在最后一行追加内容

source /etc/profile
java -version

安装Tomcat

tar xf apache-tomcat-8.0.27.tar.gz -C /application/
ln -s /application/apache-tomcat-8.0.27 /application/tomcat
echo 'export TOMCAT_HOME=/application/tomcat'>>/etc/profile
source /etc/profile
chown -R root.root /application/jdk/ /application/tomcat/

#查看安装版本
 /application/tomcat/bin/version.sh

启动

#→启动程序/application/tomcat/bin/startup.sh 

#→关闭程序/application/tomcat/bin/shutdown.sh 

cd /application/tomcat/logs/
tailf catalina.out

#查看tomcat端口
[root@zabbix-proxy ~]# netstat -lntup|grep java
tcp6       0      0 :::8080                 :::*                    LISTEN      38339/java          
tcp6       0      0 :::8009                 :::*                    LISTEN      38339/java

Tomcat日志文件

cd /application/tomcat/logs
catalina.out        tomcat实时日志,启动信息/报错信息均会在此文件体现

注: 如果catalina.out文件过大，直接清空内容。注意不能删除文件，因为程序正在使用，删除依然会继续占用系统空间，除非重启Tomcat，仅仅为了清理日志文件而重启Tomcat，这样做得不偿失。

>/application/tomcat/logs/catalina.out

Tomcat服务启动时间慢并加快访问速度？

回翻日志文件可以看到如下行：
14-Nov-2017 05:53:00.512 INFO [localhost-startStop-1] org.apache.catalina.util.SessionIdGeneratorBase.createSecureRandom Creation of SecureRandom instance for session ID generation using [SHA1PRNG] took [27,970] milliseconds.

启动时间都花费在了session ID createSecureRandom上面，这个与系统的随机数有关。

通过命令查看随机数的值
cat /proc/sys/kernel/random/entropy_avail 
注：每次使用命令“cat /dev/random”的时候就会消耗它的值。

解决方案: 安装rng-tools这个软件让其值变大
yum -y install rng-tools 
systemctl start rngd.service
cat /proc/sys/kernel/random/entropy_avail
此时可以看到值为:2989, 容量槽值就变大了，此时再重启Tomcat，速度就非常快了

<role rolename="manager-gui"/>
<role rolename="admin-gui"/>
<user username="tomcat" password="tomcat" roles="manager-gui,admin-gui"/>

Tomcat 主配置文件

文件： /application/tomcat/conf/server.xml

注：Tomcat启动会启动3个端口
8080(web访问)、8005(shutdown Tomcat)和8009
注：8009端口是(http与Tomcat通信)环境下使用，现在很少用了，可以在server.xml中禁用来停止运行

测试8005端口来关闭Tomcat

向Tomcat给8005端口发送一个“SHUTDOWN”信号即可停止Tomcat服务
echo SHUTDOWN|nc 127.0.0.1 8005
netstat -lntup |grep java   #可以看到Tomcat端口已经停止运行

Web站点部署

上线的代码有两种方式，第一种方式是直接将程序目录放在webapps目录下面。
第二种方式是使用开发工具将程序打包成war包，然后上传到webapps目录下面。

使用war包部署web站点

[root@tomcat webapps]# pwd
/application/tomcat/webapps
[root@tomcat webapps]# rz     #→上传开发提供的memtest.war
[root@tomcat webapps]# ls
docs  examples  host-manager  manager  memtest  memtest.war  ROOT

web访问：http://192.168.56.13:8080/memtest/meminfo.jsp

自定义默认站点目录

#修改服务器配置文件
vim server.xml    #在<Host></Host>字段内定义虚拟主机

新增如下内容:
<Context path="" docBase="/application/tomcat/webapps/memtest" debug="0" reloadable="false" crossContext="true"/>
<Context path="37team" docBase="/application/tomcat/webapps/memtest" debug="0" reloadable="false" crossContext="true"/>

上面两行的意思是，匹配"空"字段和“37team”字段，一旦匹配后，就指向实际站点目录/application/tomcat/webapps/memtest中。

重启Tomcat
/application/tomcat/bin/shutdown.sh
/application/tomcat/bin/startup.sh

web访问如下页面-均可以显示网页内容：
http://192.168.56.13:8080/memtest.jsp
http://192.168.56.13:8080/37team/meminfo.jsp

PS：Linux下打包压缩war和解压war包

因为种种原因需要把java程序达成war包。需要用jar命令，前提是要安装dk。

示例：

把当前目录下的所有文件打包成game.war

命令: jar -cvfM0 game.war ./


-c   创建war包
-v   显示过程信息
-f    
-M
-0   这个是阿拉伯数字，只打包不压缩的意思


解压game.war

Tomcat多实例

(1)Tomcat是单进程的，单个进程不建议部署多个站点，因为其性能是有限的，建议一个进程只部署一个站点。
(2)只会使用一个CPU核心。但如果服务器配置较高，比如:8核。这样只运行一个Tomcat实例实在浪费，所以可以通过多实例，来让服务器资源不浪费。

#复制Tomcat文件
tar -zxvf apache-tomcat-8.0.27.tar.gz 
cp -a apache-tomcat-8.0.27 tomcat8_1
cp -a apache-tomcat-8.0.27 tomcat8_2

#规避多实例端口冲突所以需要修改端口 
1. 关闭 8005  -->并修改  第1个实例改为：8011     第2个实例改为:8022
2. 关闭 8080  -->并修改  第1个实例改为：8081     第2个实例改为:8082

sed -i 's#8005#8011#;s#8080#8081#' tomcat8_1/conf/server.xml 
sed -i 's#8005#8012#;s#8080#8082#' tomcat8_2/conf/server.xml 
diff tomcat8_{1,2}/conf/server.xml  


#压缩备份实例,便于日后使用并分发
tar zcf muti_tomcat.tar.gz ./tomcat8_1 ./tomcat8_2

#拷贝修改后的示例复制到/application/
cp -a tomcat8_1 tomcat8_2 /application/

#启动多实例
/application/tomcat8_1/bin/startup.sh 
/application/tomcat8_2/bin/startup.sh 


#查看端口和进程启动情况
[root@zabbix-proxy ~]# netstat -lntup|grep java
tcp6       0      0 127.0.0.1:8011       39534/java          
tcp6       0      0 127.0.0.1:8012       39534/java

[root@zabbix-proxy ~]# ps -ef |grep java

Web访问：
http://192.168.56.13:8081/
http://192.168.56.13:8082/

Tomcat反向代理集群

用nginx代理后端的Tomcat多实例来实现负载均衡。

#安装nginx
yum install -y nginx
systemctl start nginx

nginx测试：http://192.168.56.13/

配置nginx upsteam

vim /etc/nginx/nginx.conf

 #添加upstream模块并配置location

 upstream web_pools {
        server 192.168.56.13:8081;
        server 192.168.56.13:8082;
        }

    server {
        listen       80;
        server_name  localhost;

        location / {
            index  index.jsp index.html index.htm;  
            proxy_pass http://web_pools;
        }

注： 因为页面是jsp，需要加index.jsp        

nginx -t
systemctl restart nginx

Web访问: http://192.168.56.13/

为便于看出效果追加内容：
echo 8081 >>/application/tomcat8_1/webapps/ROOT/index.jsp 
echo 8082 >>/application/tomcat8_2/webapps/ROOT/index.jsp 

再次访问-查看代理效果：
Web访问: http://192.168.56.13/

Tomcat监控

jps命令

jps命令用来输出java的状况信息，如：进程的PID值

#快速获取Java进程的pid，不用ps和grep
jps -lvm  

jps -lvm  与ps -ef|grep java结果相同

jstack命令

jstack命令主要用来查看某个java进程内的线程堆栈信息。

一旦发现Tomcat有严重异常，需要直接上脚本运行，查看结果
Tomcat Java脚本下载：https://github.com/wuyanteng/wuyanteng.github.io/tree/master/script

脚本输出结果可以看到哪些java进程占用多少资源？把结果与java开发分享一同解决问题。

比如:Tomcat运行特别卡，就使用这个查看即可。
[root@zabbix-proxy ~]# sh show-busy-java-threads.sh

使用zabbix监控Tomcat

第一步：配置Tomcat远程管理
第二步：zabbix server安装zabbix-java-gateway

1. 配置Tomcat远程管理
添加hosts解析
vim /etc/hosts     #必须添加hosts解析，否则启动失败
192.168.56.13  tomcat13

2.开启JMX远程监控
 vim /application/tomcat/bin/catalina.sh 

添加如下内容:
CATALINA_OPTS="$CATALINA_OPTS
-Dcom.sun.management.jmxremote 
-Dcom.sun.management.jmxremote.port=12345  
-Dcom.sun.management.jmxremote.ssl=false          #远程ssl验证为false
-Dcom.sun.management.jmxremote.authenticate=false #关闭权限认证
-Djava.rmi.server.hostname=192.168.56.13          #填写Tomcat IP地址

3.重启Tomcat
netstat -tunlp|grep 12345

################################
如果端口起不来，看日志
tail /application/tomcat/logs/catalina.out

4. zabbix-server上安装启动zabbix-java-gateway
yum install -y java java-devel zabbix-java-gateway
vim /etc/zabbix/zabbix_java_gateway.conf    #java配置文件
JavaGateway=192.168.56.13                   #gateway地址
StartJavaPollers=5                          #预启动进程轮训个数 

/etc/init.d/zabbix-java-gateway start
netstat -tunlp|grep java

5. 配置zabbix server支持java监控

#新增如下配置
sed -i -e '217a JavaGateway=127.0.0.1' -e '225a JavaGatewayPort=10052'  -e '235a StartJavaPollers=5' /etc/zabbix/zabbix_server.conf

#重启zabbix server

6. zabbix web配置
   配置 -- 主机 -- 添加主机 -- JMX接口: 192.168.56.13 端口:12345
   模板：选择Template JMX Generic和Template JMX Tomcat

Tomcat安全优化

安全优化

(1)端口保护: 即修改默认配置文件端口
(2)注释ajp连接端口8009, server.xml中搜索注释
(3)删除webapps目录下的默认目录，只保留ROOT空目录即可。
(4)降权启动：使用非root启动tomcat （nginx或tomcat等都支持以普通用户登录）

Tomcat降权启动

useradd tomcat
cp -a /application/tomcat8_1 /home/tomcat/
chown -R tomcat.tomcat /home/tomcat/
su -c '/home/tomcat/tomcat8_1/bin/startup.sh' tomcat
ps -ef |grep java      #可以看到启动的用户已经改成了tomcat

性能优化

Tomcat非常耗内存,并非Tomcat软件本身内存消耗大，而是运行的代码导致内存消耗大
优化方法：
1. 加大内存
2. 减少内存消耗量:开发写的代码只是不停消耗内存,但是从来进行释放
   垃圾回收机制:定期扫描内存使用情况，找到未被使用的内存,然后回收。
       特点：周期性,回收机制也会占用一定的内存使用量,同时每次回收垃圾都会使业务停止
注：网上的教程基本都是 JVM优化主要从垃圾回收策略开始阐述的
3. 简单粗暴法：定期重启Tomcat（如：定时凌晨重启Tomcat）

Tomcat默认消耗1/4系统内存，可以通过修改配置文件将阈值调到适当值。

JVM调优

vim /home/tomcat/tomcat8_1/bin/catalina.sh 
添加如下内容:
JAVA_OPTS="-Djava.awt.headless=true -Dfile.encoding=UTF-8 -server -Xms2048m -Xmx3072m -XX:NewSize=512m -XX:MaxNewSize=2048m -XX:PermSize=512m -XX:MaxPermSize=512m"

server:一定要作为第一个参数，在多个CPU时性能佳
-Xms：初始堆内存Heap大小，使用的最小内存,cpu性能高时此值应设的大一些
-Xmx：初始堆内存heap最大值，使用的最大内存
上面两个值是分配JVM的最小和最大内存，取决于硬件物理内存的大小，建议均设为物理内存的一半。
-XX:PermSize:设定内存的永久保存区域
-XX:MaxPermSize:设定最大内存的永久保存区域
-XX:MaxNewSize:
-Xss 15120 这使得JBoss每增加一个线程（thread)就会立即消耗15M内存，而最佳值应该是128K,默认值好像是512k.
+XX:AggressiveHeap 会使得 Xms没有意义。这个参数让jvm忽略Xmx参数,疯狂地吃完一个G物理内存,再吃尽一个G的swap。
-Xss：每个线程的Stack大小
-verbose:gc 现实垃圾收集信息
-Xloggc:gc.log 指定垃圾收集日志文件
-Xmn：young generation的heap大小，一般设置为Xmx的3、4分之一
-XX:+UseParNewGC ：缩短minor收集的时间
-XX:+UseConcMarkSweepGC ：缩短major收集的时间

最后重启Tomcat服务!
